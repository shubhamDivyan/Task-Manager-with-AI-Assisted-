# Code Review – Observations on AI Generated Code

While building this task manager application, I used AI tools such as Copilot to generate most of the initial code for both the backend and frontend. The generated code helped speed up development, but during the review process I noticed several issues and areas where the AI suggestions were incomplete or not following good development practices. Below are the key issues I identified and how I fixed them.

---

## 1. Missing Error Handling in API Routes

**File:** `server/server.js`

The AI generated basic CRUD routes for tasks, but it did not include proper error handling. For example, the API assumed that everything would always work correctly. In real applications, unexpected errors such as invalid input, server failures, or runtime exceptions can occur.

Originally the routes were written without any `try/catch` blocks. This means if something fails inside the route, the server could crash or return an unclear error.

**What I changed**

I wrapped the route logic inside `try/catch` blocks and returned proper error responses. This ensures that if something goes wrong, the API returns a clear message instead of failing silently.


## 2. Weak Input Validation in Task Creation

**File:** `server/server.js`

The AI generated validation that only checked if `title` existed. However, it did not handle cases where the title might be an empty string or contain only spaces.

For example, a request like this would pass validation:

```json
{
 "title": "   "
}
```

This would create an invalid task with an empty title.

**What I changed**

I improved the validation logic by trimming the input and checking that it actually contains meaningful text.

```javascript
if (!title || title.trim() === "") {
  return res.status(400).json({ error: "Title is required" });
}
```

This ensures that tasks cannot be created with empty titles.

---

## 3. No Error Handling in Frontend API Calls

**File:** `client/src/App.jsx`

The fetch requests generated by AI assumed the API would always respond successfully. There was no error handling if the network request failed or if the server returned an error.

Original example:

```javascript
const res = await fetch("/api/tasks");
const data = await res.json();
setTasks(data);
```

If the server was offline or returned an error, this would break silently.

**What I changed**

I wrapped the fetch logic inside a `try/catch` block and added a console error message.

```javascript
try {
  const res = await fetch("/api/tasks");
  const data = await res.json();
  setTasks(data);
} catch (error) {
  console.error("Failed to fetch tasks:", error);
}
```

This makes debugging easier and prevents the UI from failing silently.

---

## 4. Hardcoded API Paths

The AI generated fetch requests directly using `/api/tasks`. While this works during development, it is not flexible for production environments where the API URL might change.

**What I changed**

I suggested using environment variables for the API base URL.

Example:

```javascript
const API_BASE = backendAPI/render.com;
fetch(`${API_BASE}/tasks`)
```

This makes the application easier to configure across different environments.

---

## 5. Missing Edge Case Handling in PATCH and DELETE Routes

In the initial AI-generated code, the `PATCH` and `DELETE` routes assumed that the task ID would always exist. If a request was made with an invalid ID, the API did not handle it clearly.

**What I changed**

I added checks to verify that the task exists before updating or deleting it. If the task is not found, the API now returns a `404` response.

```javascript
if (!task) {
  return res.status(404).json({ error: "Task not found" });
}
```

This prevents unexpected behavior and improves API reliability.

---

# Summary

AI tools helped me generate the initial structure and saved a significant amount of time, especially when creating repetitive code like CRUD routes and React components. However, the generated code was not perfect and required careful review.

The main issues I found were:

* Missing error handling in both backend and frontend code
* Weak input validation
* Hardcoded values
* Missing edge case checks

After reviewing and improving the generated code, the application became more stable and closer to real-world coding standards. This exercise highlighted that AI tools are very useful for speeding up development, but developers still need to review the code carefully and apply best practices.
